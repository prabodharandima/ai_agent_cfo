# MCP Integration Current State

## Scope and conclusion

This document records the implementation observed for `TASK-MCP-REF-001`. It is based on the current API, both MCP hosts, configuration, dependency injection, health checks, and tests, plus the XML API documentation installed with `ModelContextProtocol.Core` 1.4.1.

The application already uses the official MCP SDK correctly for lazy Streamable HTTP connection, initialization, `tools/list`, and `tools/call`. It does **not** implement dynamic LLM tool selection. Discovery currently serves as an exact capability gate: the application keeps only tool names, requires the server's tool set to exactly match a hard-coded list, and then business code calls a hard-coded tool with trusted C# arguments. The SDK client connection is cached; discovered tools and schemas are not.

The switch in `CfoOrchestratorAgent.HandleAsync` is business-level intent routing. It selects a specialist agent and is not the MCP tool-selection mechanism. It should not be removed merely to make MCP invocation more dynamic.

## End-to-end call flow

1. `ChatEndpoints.HandleChatAsync` resolves `CfoOrchestratorAgent` and sends the user message through `HandleAsync`.
2. `CfoOrchestratorAgent.ClassifyAsync` asks the configured `IChatClient` for one allowed `CfoIntent`; malformed output falls back to deterministic keyword classification.
3. `CfoOrchestratorAgent.HandleAsync` uses a switch on the business intent to invoke `SalesAnalysisAgent`, `ForecastingAgent`, or `FinancialKnowledgeAgent`. `Mixed` invokes Forecasting and Knowledge, bounded to two specialists.
4. Sales and Forecasting call typed `IFinanceMcpClient` methods. The Knowledge agent may call `IKnowledgeFileMcpClient.ListFilesAsync` as an availability/access check, but semantic retrieval and citations still come from `FinancialKnowledgeRetrievalService` and ChromaDB.
5. The remote MCP client lazily creates an `HttpClientTransport` for the configured `{BaseUrl}/mcp` endpoint and calls `McpClient.CreateAsync`. The official SDK performs the MCP initialization handshake as part of that call.
6. Before every tool call, `DiscoverToolsAsync` calls SDK `ListToolsAsync`, extracts only names, and requires exact equality with the hard-coded required set.
7. A typed client method chooses a hard-coded tool name and constructs arguments in C#. Private `CallToolAsync<T>` invokes SDK `CallToolAsync` and deserializes the text result into a fixed response envelope and DTO.
8. Deterministic structured data is calculated or mapped before any LLM formatting call. Discovered MCP tools are never placed in `ChatOptions.Tools`.
9. MCP timeout, transport, capability, and response failures become `McpDependencyException`; API exception handling returns sanitized Problem Details, normally HTTP 503. Caller cancellation is rethrown unchanged.

## Dependency injection, configuration, and lifecycle

- `src/CfoAgent.Api/Program.cs` binds and validates `McpOptions`, registers named HTTP clients with infinite `HttpClient.Timeout`, and lets each MCP client enforce its configured linked-token timeout.
- `FinanceMcpClient` is one singleton exposed as both `IFinanceMcpClient` and `IFinanceMcpRemoteClient`.
- `KnowledgeFileMcpHttpClient` is the singleton `IKnowledgeFileMcpRemoteClient`; `KnowledgeFileMcpAccess` is the application-facing `IKnowledgeFileMcpClient`; `KnowledgeFileMcpClient` is the secure local fallback implementation.
- Registration is lazy. The first readiness or business operation creates each remote SDK client and performs initialization. Container health checks commonly cause this before the first chat request.
- `Mcp:Finance` and `Mcp:KnowledgeFiles` configure only enabled state, base URL, timeout, and Knowledge root/fallback. The LLM cannot choose an endpoint.
- Compose enables both remote services, points them at internal hostnames, and disables Knowledge local fallback. Neither MCP port is published to the host.
- Both SDK clients are asynchronously disposed and own their named `HttpClient` transport instance. Caller cancellation remains distinct from configured timeout.

## Error and health behavior

- `FinanceMcpClient.ExecuteDependencyOperationAsync` and `KnowledgeFileMcpHttpClient.ExecuteDependencyOperationAsync` preserve caller cancellation, map internal cancellation to `Timeout`, preserve MCP dependency failures, and map other exceptions to `Unavailable`.
- An SDK error result, empty/multiple missing text payload, unsuccessful envelope, or JSON failure becomes `InvalidResponse`.
- `McpConfigurationHealthCheck` discovers Finance tools and exercises Knowledge listing. It returns sanitized unavailable descriptions and propagates caller cancellation.
- `ApiExceptionHandler` maps all `McpDependencyException` instances to sanitized HTTP 503 Problem Details and logs dependency name/failure category without endpoint or payload details.
- Finance dependency failures propagate through Sales, Forecasting, and the orchestrator. Knowledge can fall back only in Development when explicitly configured; otherwise its agent wraps failure into a controlled unavailable operation and the API still returns a sanitized response.
